## 1. STM32 HAL Integration

### Repos cloned into `ThirdParty/`
| Repo | Purpose |
|---|---|
| `ARM-software/CMSIS_5` (sparse, Core/Include only) | `core_cm4.h` and ARM processor headers |
| `STMicroelectronics/cmsis_device_f4` | `stm32f411xe.h`, `system_stm32f4xx.c` |
| `STMicroelectronics/stm32f4xx_hal_driver` | GPIO, RCC, Cortex HAL modules |

### CMake wrapper — `ThirdParty/HAL/CMakeLists.txt`
Builds a `stm32f4xx_hal` static library with only the modules we use:
`stm32f4xx_hal.c`, `stm32f4xx_hal_cortex.c`, `stm32f4xx_hal_gpio.c`,
`stm32f4xx_hal_rcc.c`, `stm32f4xx_hal_rcc_ex.c`, `system_stm32f4xx.c`

### HAL config — `Core/Inc/stm32f4xx_hal_conf.h`
Selects only the modules needed: `HAL_GPIO_MODULE_ENABLED`, `HAL_RCC_MODULE_ENABLED`,
`HAL_CORTEX_MODULE_ENABLED`, `HAL_FLASH_MODULE_ENABLED` (header only — macros used inside
the RCC driver).

### SysTick conflict fix
Both FreeRTOS and HAL want SysTick. FreeRTOS owns it.
`HAL_InitTick()` is `__weak` in the HAL — we override it in `main.c` to do nothing:
```c
HAL_StatusTypeDef HAL_InitTick(uint32_t TickPriority)
{
    (void)TickPriority;
    return HAL_OK;
}
```
`vTaskDelay` replaces `HAL_Delay` everywhere, so the missing tick increment doesn't matter.

### Startup sequence in `main()`
```c
SystemInit();   // enables FPU, sets VTOR
HAL_Init();     // initialises HAL state (HAL_InitTick stub prevents SysTick conflict)
```

### LED rewrite — `BSP/Src/led.c`
Replaced raw register writes with:
```c
__HAL_RCC_GPIOA_CLK_ENABLE();
HAL_GPIO_Init(GPIOA, &gpio);
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET / RESET);
```

### Board correction
The original code targeted **PD12–PD15** (STM32F4 Discovery layout).
The STM32F411RE **Nucleo-64** has one user LED: **LD2 on PA5**.
Updated `led.h` to `#define LED_GREEN 5` and `led.c` to use GPIOA.

## 2. Circular Buffer with Counting Semaphores

### Design — bounded producer/consumer
```
sem_free (init = CBUF_SIZE)  — producer Takes before writing, consumer Gives after reading
sem_used (init = 0)          — producer Gives after writing,  consumer Takes before reading
```

Key insight: `xSemaphoreTake` is **atomic** — it checks and decrements in one uninterruptible
operation. A plain `if (count > 0) count--` would race between two tasks.

Single producer / single consumer means no mutex is needed on `head` or `tail` — each index
is only ever touched by one side. The semaphore Give/Take also acts as a memory barrier on
Cortex-M4.

### Files
- `Core/Inc/cbuf.h` — `CircBuf_t` struct, `cbuf_init`, `cbuf_write`, `cbuf_read`
- `Core/Src/cbuf.c` — implementation

### Demo behaviour
- Producer writes ON/OFF commands every **100 ms** (10 items/sec)
- Consumer reads one command every **500 ms** (2 items/sec)
- 8-slot buffer fills after ~800 ms → producer blocks on `sem_free`
- LD2 blinks at producer rate initially, then settles to consumer rate (500 ms)

### `FreeRTOSConfig.h` changes
```c
#define configUSE_COUNTING_SEMAPHORES   1   // was 0
#define configUSE_MUTEXES               1   // was 0
```

## 3. GDB + OpenOCD Debugging

### Enable FreeRTOS thread awareness in OpenOCD
Add `-c "stm32f4x.cpu configure -rtos FreeRTOS"` to the OpenOCD command.

Newer FreeRTOS removed `uxTopUsedPriority` which OpenOCD looks for. Add a shim in `main.c`:
```c
const volatile uint32_t uxTopUsedPriority = configMAX_PRIORITIES - 1;
```

### Correct GDB session workflow
```
gdb-multiarch build/firmware.elf       # load symbols at startup

(gdb) target remote localhost:3333
(gdb) load                             # flash binary + registers symbols with OpenOCD
(gdb) monitor reset halt               # restart to clean state
(gdb) continue                         # let FreeRTOS scheduler start
Ctrl+C                                 # halt once LEDs are blinking
(gdb) info threads                     # lists all FreeRTOS tasks
```

### Board LEDs explained
| LED | Controlled by | Behaviour |
|---|---|---|
| LD1 (COM) | ST-LINK MCU | Blinks during SWD communication — unaffected by `monitor halt` |
| LD2 (USER) | STM32F411 (your code) | PA5 — the one your tasks drive |
| LD3 (PWR) | Hardware | Always solid red when board is powered |

### Useful GDB commands for this project
```
info threads                                  # list all FreeRTOS tasks + state
thread 3 / bt                                 # switch task, show call stack
print g_cbuf                                  # inspect circular buffer
print g_cbuf.head / g_cbuf.tail               # read/write indices
x/8ub g_cbuf.data                             # dump buffer contents
watch g_cbuf.head                             # break when producer writes
print uxQueueMessagesWaiting(g_cbuf.sem_free) # semaphore count
print xFreeBytesRemaining                     # heap space remaining
info registers                                # CPU registers
```

### What the debugger showed
When halted mid-run, `info threads` confirmed the design was working:
- **IDLE** — running (no user task needed CPU)
- **Consumer** — sleeping inside `vTaskDelay(500)`
- **Producer** — blocked in `xQueueSemaphoreTake` with `xTicksToWait=4294967295` (`portMAX_DELAY`)

The producer's backtrace: `producer_task → cbuf_write → xQueueSemaphoreTake` — stuck waiting
for `sem_free` to become non-zero, exactly as designed.
